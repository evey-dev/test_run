Changes summary
===============

This file records implementation and notebook decisions that are not obvious from
the code alone.


Core code changes
-----------------

`src/attribution_graph.py`

- Added `--contrast-target`.
- When provided, the attribution objective is:

    target logit - contrast target logit

- This is important for arithmetic carry because the question is whether the
  model prefers the correct carry digit over a dropped-carry alternative.
- Graph JSON records the target, contrast target, and objective metadata.

`src/intervention.py`

- Added graph-feature sign filtering:

    --graph-feature-sign positive|negative|all

- Added edit strength:

    --edit-strength

- Added full knockout component selection:

    --knockout-component mlp|attn|block

- Added broader diagnostics:

    --layer-scan
    --position-scan
    --positions last|all|comma-separated indices
    --print-tokens

- Added raw MLP swap support:

    --raw-mlp-swap

- Improved diagnostics so interventions print token positions and pre-knockout
  activation norms. This was added because it was initially unclear whether the
  code was editing the intended positions or whether activations were nonzero.

`src/train.py`

- Training now keeps and saves the best validation checkpoint rather than blindly
  saving the final epoch.

SAE training decisions
----------------------

The project brief asks for lightweight SAEs on a few selected layers. It does not
require every layer.

Final selected layers:

  4, 8, 12, 16, 20, 24, 28

Behaviour-specific final training was used because training was fast enough and
because mixed/general checkpoints made interpretation less clean.

Final config files:

- `configs/sae_units_final_train_config.yaml`
- `configs/sae_math_final_train_config.yaml`
- `configs/sae_capitals_final_train_config.yaml`


Important conceptual decisions
------------------------------

Arithmetic must be per digit.

For `58 + 83 = 141`, the tens digit is studied with:

  Question: What is 58 + 83? Answer: 1

The correct next token is `4`; the dropped-carry alternative is `3`.

The graph should use:

  --target "4" --contrast-target "3"

because this studies the carry-vs-dropped-carry decision directly.

All-position edits matter.

Final-token-only MLP interventions mostly did not affect the math output. The
stronger result came from all-position MLP knockout, which implies the relevant
state is distributed across prompt positions.

Sparse SAE features did not isolate the carry.

The current results support the existence of a broad activation-level carry
signal, but not a clean compact SAE feature group that controls it.


Current empirical story
-----------------------

Physics units:

- Best positive reproduction.
- Use as the main successful behaviour.
- Results are better aligned with intervention/swap validation than the other
  behaviours.

Math carry:

- Partial reproduction.
- Clean model strongly predicts correct digit `4`.
- Sparse SAE interventions weakly move the dropped-carry `3` upward.
- All-position MLP knockout produces the strongest causal shift:
  - `4` probability drops to about 0.6406
  - `3` probability rises to about 0.1836
  - `2` also rises, so this is broad digit destabilization, not a clean carry
    feature.

Capitals:

- Weak/negative reproduction.
- Dallas/Oakland prompts were low confidence.
- Higher-confidence controls were added but still did not produce a clean swap.
- Use as evidence that factual recall was not cleanly captured by this setup.


Report implication
------------------

The report should not claim a full Anthropic-style circuit reproduction.

The strongest claim is:

  The independent pipeline reproduces the methodology and finds some causal
  structure, especially for units and broad arithmetic interventions, but the
  lightweight SAE setup does not consistently recover compact sparse circuits in
  Qwen3-4B-Instruct.

This is consistent with the project brief because the brief values rigorous
reproducibility investigation, including partial or negative reproductions.


Final validation and report figures
-----------------------------------

- Added `run_gpu_final_validation.ipynb` as a short post-hoc validation notebook.
  It restores existing final artifacts; it does not retrain SAEs or regenerate
  attribution graphs.
- Added `src/sae_diagnostics.py` to measure held-out reconstruction fidelity,
  achieved sparsity, dead features and encoder/decoder norm distributions.
- Added `src/heldout_validation.py` to evaluate fixed graph-derived edits on 12
  new arithmetic pairs and 12 new units pairs, with broad latent/raw controls and
  deterministic bootstrap summaries.
- Added `src/plot_validation.py` for SAE-diagnostic and graph-held-out figures.
- Added `src/plot_attribution_graph.py` for a compact backward-connected view of
  a complete saved graph. The printed view is explicitly labelled as a subset.
- Added four checked vector figures under `report/figures/` and inserted them in
  `report/report.tex` in place of placeholders.
- Added `report/FIGURE_PLAN.txt` with exact filenames, data provenance, captions,
  insertion locations and rules for the three figures pending the final run.
- Corrected the final arithmetic graph filename in the README to
  `math_final_carry_58_83_4v3_graph.json`.

Scientific interpretation of the new checks:

- Weak reconstruction can be investigated as one contributor to weak sparse
  interventions, but reconstruction metrics cannot prove causation.
- Strong reconstruction with weak causal effects would make a simple
  reconstruction-error explanation untenable.
- The new benchmark is graph-held-out. Unit sources may occur in the SAE
  validation split, so they must not all be described as fully SAE-held-out.
